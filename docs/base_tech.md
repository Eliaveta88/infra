1. GastroRoute_identity (Авторизация и доступ)
Отвечает за безопасность и маршрутизацию прав.

PostgreSQL:

Таблицы: users (хэши паролей, контакты), roles, permissions (связка ролей и конкретных прав, например can_approve_credit).

Redis (Не только кэш):

Blacklist токенов: Хранение отозванных JWT (при логауте или смене пароля) с TTL, равным времени жизни токена.

Rate Limiting: Ограничение количества попыток входа (защита от брутфорса).

Активные сессии: Хранение session_id для принудительного разлогинивания пользователя со всех устройств админом.

Основные Endpoints:

POST /auth/login — выдача Access и Refresh токенов.

POST /auth/logout — занесение токена в Redis blacklist.

GET /users/me — получение профиля и списка пермишенов.

POST /users — создание сотрудника (только для админа).

2. GastroRoute_catalog (Управление номенклатурой)
Центральный справочник продуктов питания.

PostgreSQL:

Таблицы: products, categories, units (штуки, коробки, паллеты).

Особенность: Использование поля JSONB для хранения специфичных атрибутов (БЖУ, жирность, ГОСТ), чтобы не плодить десятки колонок.

Redis:

Автокомплит поиска: Хранение отсортированных множеств (Sorted Sets) для мгновенного поиска товаров по названию или артикулу в строке поиска.

Кэш каталога: Горячие данные о часто просматриваемых товарах с инвалидацией при обновлении в БД.

Основные Endpoints:

GET /products — список товаров с пагинацией и фильтрами.

GET /products/{id} — полная карточка товара.

POST /products/{id}/units — добавление коэффициентов пересчета (например, 1 коробка = 12 шт).

3. GastroRoute_warehouse (Склад и остатки)
Самый нагруженный сервис. Реализует партионный учет и алгоритм FEFO.

PostgreSQL:

Таблицы: inventory_batches (партии со сроками годности expiry_date), warehouse_cells (адресное хранение), stock_movements (история всех движений: приход, расход, списание).

Redis (Критическая логика):

Distributed Locks (Распределенные блокировки): Использование паттерна Redlock. Когда два менеджера одновременно пытаются зарезервировать последнюю коробку сыра, Redis блокирует этот SKU на миллисекунды, чтобы транзакция в БД не ушла в минус.

Горячие остатки: Агрегированное количество доступного товара для мгновенного отображения на фронтенде без тяжелых COUNT/SUM запросов к PostgreSQL.

Основные Endpoints:

POST /stock/reserve — бронирование партий под заказ (с учетом ближайшего срока годности).

POST /stock/release — снятие резерва (если заказ отменен).

POST /stock/receive — оприходование новой партии товара.

4. GastroRoute_finance (Взаиморасчеты)
Учет денег, долгов и лимитов клиентов.

PostgreSQL:

Таблицы: client_accounts (баланс контрагента), transactions (проводки), invoices (счета). Строгое использование типа NUMERIC/DECIMAL для всех сумм.

Redis:

Idempotency Keys: При создании транзакции (оплаты) с фронта передается уникальный ключ. Redis хранит его 24 часа. Это защищает от двойного списания, если клиент дважды нажал кнопку или сеть моргнула.

Основные Endpoints:

GET /accounts/{client_id} — получение текущего сальдо и кредитного лимита.

POST /transactions — регистрация платежа.

POST /invoices/generate — формирование инвойса (можно возвращать ссылку на сгенерированный PDF).

5. GastroRoute_logistics (Маршруты и доставка)
Управление транспортом и распределение грузов.

PostgreSQL:

Таблицы: vehicles (машины с ВГХ — тоннаж, объем, тип кузова), routes (маршрутные листы), route_points (точки выгрузки).

Redis:

Geospatial (GEO): Если в будущем планируется отслеживание машин, Redis отлично хранит и обновляет координаты водителей (команды GEOADD, GEORADIUS).

Pub/Sub: Мгновенное уведомление диспетчера (через WebSockets), когда водитель нажал в приложении «Заказ доставлен».

Основные Endpoints:

POST /routes/plan — создание нового рейса.

PUT /routes/{route_id}/assign — добавление заказа в маршрут (с проверкой, что вес заказа не превышает свободную грузоподъемность фургона).

PATCH /routes/points/{point_id}/status — обновление статуса доставки.

6. GastroRoute_frontend (Flutter Web)
Архитектура клиентской части для SPA (Single Page Application).

Архитектурный паттерн: Feature-First (Clean Architecture). Папки во Flutter должны зеркалить твои микросервисы (lib/features/catalog, lib/features/warehouse и т.д.).

State Management: Рекомендую Riverpod или BLoC. В сложном B2B проекте Riverpod даст отличную производительность и легкость при работе с асинхронными запросами.

Сеть: Использовать пакет Dio с настроенными интерцепторами (Interceptors) для автоматического прикрепления JWT токена ко всем запросам и прозрачной логики Refresh токена (если 401 ошибка — незаметно обновить токен и повторить запрос).

Роутинг: GoRouter. Поддерживает глубокие ссылки (Deep Links), что важно для Web-версии (чтобы менеджер мог скинуть ссылку на конкретный заказ коллеге: crm.com/orders/12345).
