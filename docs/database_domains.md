# Базы данных и границы доменов

## Важно: это не «одна база на всех»

Используется **один процесс PostgreSQL** (один контейнер `postgres`) — так проще в dev: один порт `5432`, один бэкап тома.

Внутри него созданы **отдельные логические базы** (`CREATE DATABASE …`). У каждой свой набор таблиц, свой Alembic, своё подключение у сервиса. **Кросс-схемных FK между базами в PostgreSQL нет** — связи только по **идентификаторам** (например `product_id`, `order_id`) на уровне приложения.

Учётка `gastro` / пароль в compose **одинаковые для всех БД** — это только удобство локальной разработки, не смешивание данных.

## Карта: сервис → база → бизнес-логика

| Микросервис | Имя БД (`DATABASE_URL` …/`<имя>`) | Домен данных |
|-------------|-----------------------------------|--------------|
| **catalog** | `catalog` | Каталог: товары, SKU, категории, цены |
| **finance** | `finance` | Деньги: счета клиентов, транзакции, инвойсы |
| **identity** | `identity` | Пользователи, аутентификация, сессии/токены (в БД) |
| **warehouse** | `warehouse` | Склад: партии, остатки, резервы под заказы |
| **logistics** | `logistics` | Доставка: маршруты, точки, привязка к заказам |
| **orders** | `orders` | Заказы: шапка заказа, позиции, история статусов |

**Redis** один на все сервисы — кэш, блокировки, idempotency; это не «база домена», а инфраструктурный слой.

## Почему может казаться, что «везде одинаково»

- Один хост: `postgres:5432` в Docker-сети или `localhost:5432` с хоста.
- Один пользователь БД: `gastro` (и один пароль в `.env`/compose).
- В DBeaver при одном подключении видно **список баз** — нужно переключать **Database**, это разные схемы данных.

Имена баз **разные** (`catalog`, `finance`, …) — разделение по бизнесу уже заложено.

## Если нужна жёсткая изоляция (отдельный Postgres на сервис)

Возможно вынести **6 контейнеров PostgreSQL** (или managed БД в облаке) — отдельные тома, отдельные пользователи, отдельные секреты. Плюсы: изоляция, отдельные лимиты. Минусы: больше RAM/диска, сложнее локальный стек.

Текущая схема (один Postgres, много БД) — нормальный компромисс для микросервисов на одной команде/репозитории.

## Где это в коде

- Создание БД: `docker/postgres/init-databases.sql`
- URL у каждого сервиса в `docker-compose.yml`: путь заканчивается на `/catalog`, `/finance`, и т.д.
